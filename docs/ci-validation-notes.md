# CI Validation Notes

## Goal

This document records how CI validates the repository during the stabilization phase.

## Current Workflow

Workflow file:

```text
.github/workflows/go-test.yml
```

Current triggers:

```text
push to main
pull_request to main
manual workflow_dispatch
```

Current validation sequence:

```text
1. checkout
2. setup Go 1.22
3. go mod tidy
4. git diff --exit-code -- go.mod go.sum
5. go test ./...
```

## Why go mod tidy Runs Before Tests

The workflow runs `go mod tidy` before tests to ensure module metadata is reproducible.

During PR-037, `gopkg.in/yaml.v3` was deferred and `go.mod` was restored to no external dependencies.

The stabilization workflow should still fail if:

```text
- go.mod changes after tidy
- go.sum changes after tidy
- tests do not compile or pass
```

If a future PR reintroduces `gopkg.in/yaml.v3`, `go.sum` must be generated by Go tooling and committed.

## Important Interpretation

A failure in the `Verify module files are tidy` step means dependency metadata is not committed or not reproducible.

A failure in the `Run tests` step means compile or test execution failed after module files are already tidy.

An empty visible status result from the current connector means CI state is unknown. It is not a pass and not a fail.

## Manual Validation Path

The workflow now supports manual dispatch.

Use GitHub Actions UI:

```text
Actions → Go Test → Run workflow → main
```

Then inspect the `Verify module files are tidy` and `Run tests` steps.

## Current Boundary

This CI does not perform:

```text
- integration tests against real OpenAI-compatible providers
- Kubernetes API server tests
- GitHub PR creation tests
- kubectl apply
- controller-runtime tests
```

## Required Local Commands

Run locally before pushing stabilization changes:

```bash
go mod tidy
go test ./...
```

Commit `go.sum` only if `go mod tidy` generates it.

Do not hand-write `go.sum` hashes.
