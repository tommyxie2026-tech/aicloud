# AI Cloud v0.1 Engineering Milestones

## Roadmap Goal

AI Cloud v0.1 must prove that the platform can execute governed AI tasks across multiple model deployment modes without depending on one provider.

The engineering path prioritizes:

1. unified model protocol and Model Registry;
2. hybrid commercial and private model access;
3. task-level cost accounting;
4. agent observability and continuous evaluation;
5. secure Tool Gateway and Policy Engine boundaries;
6. license and model-supply-chain governance;
7. capacity-aware routing and graceful degradation.

The v0.1 success unit is not a successful model API call. It is a successful, traceable, policy-compliant task with measurable quality, cost, and reliability.

## Milestone 1: Unified Model Core and Registry

Deliver:

- Go service skeleton;
- provider-neutral ModelProvider interface;
- normalized request, response, tool-call, error, and usage schemas;
- Model Registry;
- model version and lifecycle state;
- capability metadata;
- deployment-mode metadata;
- pricing metadata;
- license and risk metadata fields;
- deterministic Mock Provider;
- at least two real provider adapters, including one commercial and one private or self-hosted path.

Acceptance criteria:

- the same task request can be routed to different providers without changing the platform API;
- provider SDK types remain inside adapters;
- every response records model identity, model version, provider, deployment mode, token usage, latency, and trace ID;
- the Registry can distinguish active, degraded, deprecated, and retired models.

## Milestone 2: Task Runtime, Trace, and Cost Ledger

Deliver:

- Task API and task repository;
- Agent interface;
- Temporal workflow;
- task state machine;
- model invocation activities;
- task event stream;
- OpenTelemetry trace hierarchy;
- usage and cost-event model;
- task-level cost aggregation;
- retry and failure accounting.

Cost dimensions:

- tenant;
- project;
- user or service account;
- agent;
- task;
- workflow run;
- model;
- tool;
- sandbox.

Acceptance criteria:

- each task exposes selected model, selection reason, token use, retries, latency, total cost, and final status;
- model, tool, workflow, and sandbox costs can be aggregated into cost per successful task;
- API, workflow, model, tool, policy, and sandbox spans share one trace ID;
- failed and retried attempts remain visible rather than being hidden from cost reporting.

## Milestone 3: Secure Hybrid Execution

Deliver:

- Tool Registry;
- Tool Gateway;
- initial Git, GitHub, filesystem, and shell tools;
- OPA policy integration;
- Credential Broker interface;
- short-lived credential path;
- Sandbox API;
- Kubernetes Job-based sandbox;
- CPU, memory, execution-time, filesystem, and network limits;
- approval state and human-approval callback;
- immutable audit events.

Mandatory execution path:

```text
Agent
  |
Tool Gateway
  |
Policy Engine
  |
Credential Broker
  |
Sandbox or Enterprise Resource
```

Acceptance criteria:

- an agent cannot call enterprise systems directly;
- denied tool actions are blocked before credential issuance;
- high-risk actions can pause in WAITING_APPROVAL and resume safely;
- all tool calls record actor, task, policy decision, credential scope, input digest, result, and trace ID;
- sandbox cleanup is deterministic after success, failure, cancellation, or timeout.

## Milestone 4: Capacity-Aware Routing and Graceful Degradation

Deliver:

- endpoint health checks;
- provider quota and rate-limit state;
- latency and error-rate signals;
- capability-aware routing;
- tenant-policy-aware routing;
- cost-budget-aware routing;
- timeout and bounded retry;
- circuit breaker;
- ordered fallback chains;
- admission control;
- degraded-mode response policy;
- cache-isolation rules.

Acceptance criteria:

- outage, timeout, or quota exhaustion on the primary provider triggers an allowed fallback;
- fallback never violates tenant policy, data-residency policy, or model capability requirements;
- a task records why each model was selected, rejected, retried, or replaced;
- no cross-tenant prompt, response, embedding, or tool-result cache reuse occurs;
- the platform can reject work before execution when no safe capacity is available.

## Milestone 5: Evaluation and Model Supply-Chain Governance

Deliver:

- evaluation dataset format;
- benchmark runner;
- task-quality scoring;
- latency, cost, safety, stability, and human-intervention metrics;
- model and prompt regression comparison;
- release-gate policy;
- production trace sampling;
- model license-evidence records;
- upstream model and dataset provenance fields;
- model artifact digest and signature fields;
- model security and approval status;
- production-routing eligibility check.

Acceptance criteria:

- a model or prompt upgrade can be compared with the current production version on the same task set;
- production rollout can be blocked by quality, safety, cost, or reliability regression;
- models without required license or supply-chain approval cannot enter production routing;
- evaluation results are linked to exact model, prompt, workflow, tool, and dataset versions;
- public benchmark scores are treated as reference data, not as the sole routing criterion.

## Milestone 6: Enterprise Governance

Deliver:

- tenant and project hierarchy;
- RBAC;
- workload identity;
- tenant and project budgets;
- model allow and deny policies;
- regional and data-residency policies;
- showback reports;
- audit-query API;
- capacity quotas;
- provider and model ownership metadata.

Acceptance criteria:

- cost, usage, quality, and audit data can be reported by tenant, project, user, agent, task, model, and tool;
- each model and tool has a responsible owner;
- budget policy can warn, throttle, reroute, or reject work;
- one tenant cannot observe or influence another tenant's resources, traces, cache entries, credentials, or budgets.

## Milestone 7: Developer AI Cloud MVP

Scenario:

```text
GitHub Issue
    |
Code Agent
    |
Model Router
    |
Workflow
    |
Policy and Approval
    |
Sandbox and Tools
    |
Code Change
    |
Test and Evaluation
    |
Pull Request
```

Success criteria:

- multiple model providers behind one protocol;
- at least one commercial and one self-hosted or private deployment path;
- secure agent execution;
- complete trace and audit record;
- measurable total task cost and cost per successful task;
- reproducible evaluation;
- license-approved model selection;
- primary-provider failure handled through a policy-compliant fallback;
- generated pull request includes task summary, tests, selected model, cost, and trace reference;
- a human can review and approve all high-risk actions before execution.

## v0.1 Priority Gates

### Gate A: Model Independence

Milestone 2 cannot be considered complete until task execution is provider-neutral.

### Gate B: Task Economics

Milestone 3 cannot be considered complete until all execution components emit cost events.

### Gate C: Traceability

Milestone 4 cannot be considered complete until routing and fallback decisions are traceable.

### Gate D: Trust and Compliance

Milestone 7 cannot enter production demonstration until tool access, model license status, policy decisions, and approvals are auditable.

## Deferred Beyond v0.1

The following remain outside the critical v0.1 path:

- foundation-model training;
- general GPU-cluster management;
- advanced agent marketplace monetization;
- fully autonomous production changes;
- multi-region optimization beyond basic failover;
- automated ROI optimization;
- broad industry-template marketplace.

These should not delay the first complete, governed Developer AI Cloud workflow.
