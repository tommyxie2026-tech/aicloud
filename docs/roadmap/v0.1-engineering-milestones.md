# AI Cloud v0.1 Engineering Milestones

## Roadmap Goal

AI Cloud v0.1 must prove that the platform can execute governed AI tasks across multiple model deployment modes without depending on one provider.

The engineering path prioritizes:

1. unified model protocol and an operational Model Registry;
2. deterministic, efficient, specialist and flagship routing paths;
3. hybrid commercial and private model access;
4. task-level cost accounting;
5. secure Tool Gateway and Sandbox execution before broad Agent autonomy;
6. agent observability and reproducible continuous evaluation;
7. license and model-supply-chain governance based on evidence;
8. capacity-aware routing and graceful degradation.

The v0.1 success unit is not a successful model API call. It is a successful, traceable, policy-compliant task with measurable quality, cost, safety and reliability.

For the detailed 2026-07-24 sequencing adjustment, see `2026-07-24-market-driven-priority-adjustment.md`.

## Milestone 1: Operational Model Core and Registry

Deliver:

- Go service skeleton;
- provider-neutral `ModelProvider` interface;
- normalized request, response, tool-call, error and usage schemas;
- deterministic Mock Provider;
- at least two real provider adapters, including one commercial and one private or self-hosted path;
- Model Registry with immutable model-version records;
- model lifecycle state;
- capability and specialist-task metadata;
- deployment-mode and data-residency metadata;
- pricing and self-hosted allocation metadata;
- runtime health, latency, error-rate, quota, capacity and queue signals;
- evaluation-suite and evidence-version references;
- license, provenance, artifact-digest, signature, risk and approval metadata.

Acceptance criteria:

- the same task request can be routed to different providers without changing the platform API;
- provider SDK types remain inside adapters;
- every response records model identity, model version, provider, deployment mode, token usage, latency and trace ID;
- the Registry distinguishes draft, active, degraded, deprecated, retired and revoked model versions;
- stale runtime signals are identifiable and cannot be treated as current capacity;
- unapproved or revoked model versions cannot enter production routing.

## Milestone 2: Routing Core and Task Cost Ledger

Deliver:

- route classes for deterministic, efficient, specialist and flagship execution;
- `RouteDecision`, `RouteCandidate` and `FallbackPolicy` schemas;
- capability-, evaluation-, policy-, budget-, residency- and license-aware candidate filtering;
- model and provider health input;
- route-reason and rejected-candidate recording;
- Task API and task repository;
- usage and immutable `CostEvent` model;
- model-call cost calculation;
- task-level cost aggregation foundation;
- predicted cost before execution;
- retry and failure accounting.

Cost dimensions:

- tenant;
- project;
- user or service account;
- agent;
- task;
- workflow run;
- model;
- tool;
- sandbox.

Acceptance criteria:

- each task exposes selected route, selected model, selection reason, rejected alternatives, token use, retries, latency, estimated cost and final status;
- deterministic and model-backed routes use one Task interface;
- model-call costs can be aggregated into cost per successful task;
- failed and retried attempts remain visible rather than being hidden from cost reporting;
- a routing decision can be replayed from stored candidate inputs and evidence versions.

## Milestone 3: Secure Tool and Sandbox Execution

Deliver:

- Tool Registry;
- Tool Gateway;
- initial Git, GitHub, filesystem and restricted-shell tools;
- OPA policy integration;
- Credential Broker interface;
- task-scoped short-lived credential path;
- Sandbox API;
- Kubernetes Job-based sandbox;
- namespace and service-account isolation;
- CPU, memory, execution-time, filesystem and network limits;
- network deny by default;
- approval state and human-approval callback;
- immutable audit events;
- deterministic cleanup after success, failure, cancellation or timeout.

Mandatory execution path:

```text
Agent
  |
Tool Gateway
  |
Policy Engine
  |
Approval when required
  |
Credential Broker
  |
Sandbox or Approved Enterprise Resource
```

Acceptance criteria:

- an Agent cannot call enterprise systems directly;
- denied tool actions are blocked before credential issuance;
- high-risk actions can pause in `WAITING_APPROVAL` and resume safely;
- all tool calls record actor, task, policy decision, credential scope, input digest, result and trace ID;
- sandbox execution has resource, identity, network and duration boundaries;
- sandbox cleanup is deterministic.

## Milestone 4: Durable Task Runtime, Trace and Full Economics

Deliver:

- Agent interface;
- Temporal workflow;
- complete task state machine;
- model invocation activities;
- Tool and Sandbox activities;
- durable retry, timeout, cancellation and resume;
- task event stream;
- OpenTelemetry trace hierarchy;
- model, tool, workflow, sandbox, storage and network cost events;
- reconciled task-level cost report;
- human-approval cost field where measurable.

Acceptance criteria:

- a task survives API server or worker restart;
- workflow state and retries are deterministic;
- approval pauses and resumes the same task;
- API, workflow, model, tool, policy, sandbox and evaluation spans share one trace ID;
- every completed or failed Task has a reconciled cost breakdown;
- cost per successful task is queryable.

## Milestone 5: Evaluation Evidence and Specialist Routing

Deliver:

- evaluation dataset format;
- benchmark runner;
- task-quality scoring;
- task-class taxonomy;
- specialist capability labels for code, security, structured extraction, multimodal and other initial domains;
- versioned evaluation configuration;
- exact model, prompt, workflow, tool, permission, token-budget, time-budget, retry, compaction, sandbox and dataset references;
- raw output and score retention;
- latency, cost, safety, stability and human-intervention metrics;
- model and prompt regression comparison;
- release-gate policy;
- production trace sampling;
- evaluation-driven specialist, efficient and flagship routing.

Acceptance criteria:

- a model or prompt upgrade can be compared with the current production version on the same task set and configuration;
- production rollout can be blocked by quality, safety, cost, latency or reliability regression;
- evaluation results are linked to exact model, prompt, workflow, tool and dataset versions;
- a specialist model is preferred only when evidence is relevant to the current task class;
- public benchmark scores are treated as reference data, not as the sole routing criterion.

## Milestone 6: Capacity-Aware Routing and Graceful Degradation

Deliver:

- endpoint health checks;
- provider quota and rate-limit state;
- available concurrency, queue depth and signal-freshness state;
- latency and error-rate windows;
- timeout and bounded retry;
- circuit breaker;
- ordered, policy-filtered fallback chains;
- admission control;
- degraded-mode response policy;
- deterministic non-model fallback where applicable;
- cache-isolation rules.

Acceptance criteria:

- outage, timeout, overload or quota exhaustion on the primary route triggers an allowed fallback;
- fallback never violates tenant policy, data-residency policy, license status, budget or model capability requirements;
- a task records why each model was selected, rejected, retried or replaced;
- added fallback latency and cost are recorded;
- no cross-tenant prompt, response, embedding or tool-result cache reuse occurs;
- the platform rejects or queues work when no safe capacity is available.

## Milestone 7: Evidence-Based Model Supply-Chain Governance

Deliver:

- model license-evidence records;
- authoritative license-text or reference storage;
- upstream base-model and dataset provenance fields;
- commercial-use, redistribution, hosted-service and notice restrictions;
- model artifact digest and signature fields;
- model security scan status;
- reviewer and approval records;
- lifecycle states: discovered, evidence-collected, reviewed, approved, restricted, rejected and revoked;
- production-routing eligibility check;
- immediate revocation behavior.

Acceptance criteria:

- a license label alone is not sufficient for production admission;
- models without required license, provenance, security or approval evidence cannot enter production routing;
- routing checks the immutable model-version approval state at execution time;
- revoked models are removed from new routing without breaking historical auditability;
- historical Tasks remain linked to the exact immutable model version used.

## Milestone 8: Enterprise Governance

Deliver:

- tenant and project hierarchy;
- RBAC;
- workload identity;
- tenant and project budgets;
- model allow and deny policies;
- regional and data-residency policies;
- showback reports;
- audit-query API;
- capacity quotas;
- provider, model and tool ownership metadata.

Acceptance criteria:

- cost, usage, quality and audit data can be reported by tenant, project, user, agent, task, model and tool;
- each model and tool has a responsible owner;
- budget policy can warn, throttle, reroute or reject work;
- one tenant cannot observe or influence another tenant's resources, traces, cache entries, credentials or budgets.

## Milestone 9: Developer AI Cloud MVP

Scenario:

```text
GitHub Issue
    |
Task Classification
    |
Deterministic / Efficient / Specialist / Flagship Router
    |
Agent Planning
    |
Policy and Approval
    |
Tool Gateway and Sandbox
    |
Code Change
    |
Test and Evaluation
    |
Pull Request
    |
Trace, Route and Cost Report
```

Success criteria:

- multiple model providers behind one protocol;
- at least one commercial and one self-hosted or private deployment path;
- deterministic execution path for supported simple work;
- secure Agent execution;
- complete trace and audit record;
- measurable total task cost and cost per successful task;
- reproducible evaluation evidence;
- specialist routing based on task-relevant evidence;
- license-approved model selection;
- primary-route failure handled through a policy-compliant fallback;
- generated Pull Request includes task summary, tests, selected route and model, fallback status, cost and trace reference;
- a human can review and approve all high-risk actions before execution.

## v0.1 Priority Gates

### Gate A: Model Independence and Operational Registry

Milestone 2 cannot be considered complete until task execution is provider-neutral and routing reads governed operational metadata from the Model Registry.

### Gate B: Task Economics

Milestone 4 cannot be considered complete until model, tool, workflow and sandbox components emit cost events.

### Gate C: Secure Execution

Broad Agent functionality cannot be promoted to production status until Tool Gateway, policy, short-lived credentials and Sandbox controls are enforced.

### Gate D: Evaluation Reproducibility

Specialist or evaluation-driven routing cannot be enabled until the complete evaluation configuration is versioned and retained.

### Gate E: Reliability

The Developer MVP cannot be considered reliable until routing and fallback decisions are traceable and tested under outage, quota and overload conditions.

### Gate F: Trust and Compliance

The Developer MVP cannot enter production demonstration until tool access, model license evidence, supply-chain approval, policy decisions and human approvals are auditable.

## Current Implementation Status

The first Skeleton Code implementation is present in Draft PR #1 on branch `agent/aicloud-v01-skeleton`.

Implemented, pending merge:

- Go API and worker entrypoints;
- modular internal packages for control plane, model service, Agent Runtime, workflow, Tool Gateway, policy, configuration, logging, telemetry and repositories;
- in-memory Model and Task repositories plus a PostgreSQL migration contract;
- health/readiness and minimal Model/Task HTTP APIs;
- Dockerfile, PostgreSQL/Redis Compose services and a minimal Helm chart;
- unit coverage for the HTTP API.

Not yet implemented:

- PostgreSQL runtime adapters;
- real provider routing;
- immutable cost ledger;
- Temporal workflow execution;
- OPA/Rego runtime integration;
- Kubernetes Sandbox;
- OpenTelemetry SDK wiring;
- evaluation evidence chain;
- capacity-aware fallback;
- license-evidence admission;
- authentication, multi-tenancy and production RBAC.

The repository remains a modular monolith. The immediate engineering goal is to merge and stabilize the skeleton, then implement operational Registry fields, real provider routing and model-call cost events before expanding Agent behavior.

## Deferred Beyond v0.1

The following remain outside the critical v0.1 path:

- foundation-model training;
- general GPU-cluster management;
- advanced Agent marketplace monetization;
- fully autonomous production changes;
- multi-region optimization beyond basic failover;
- automated ROI optimization;
- broad industry-template marketplace.

These should not delay the first complete, governed Developer AI Cloud workflow.
